# 回顧——PM 學 Claude Code，到底學到了什麼

## 寫到這裡，回頭看看自己走了多遠

寫前面幾篇的時候，我常常邊寫邊想：這些事當時發生的時候，明明覺得自己手忙腳亂、什麼都不懂，怎麼現在回頭看，好像也沒那麼難？

從一開始連 repo、remote 是什麼都搞不清楚，到後來能自己判斷「這個工具該複製還是該共用」，中間其實沒有什麼戲劇性的轉折點，就是一次又一次遇到問題、問清楚、動手處理，慢慢累積出來的。這篇想把這些累積下來的東西，整理成幾個真正重要的心得。

## 五個核心心得：PM 學 Claude Code 帶走了什麼？

**這個系列從 Git 基礎概念出發，經過資料夾整理、資安處理、工作流程模板化，到跨專案工具管理，五個心得環環相扣，構成一套非工程背景 PM 可以直接套用的 AI 工作流。**

1. **懂一點 Git／GitHub 的基本邏輯，是跟 AI agent 一起工作的基礎**——不需要會寫程式，但要懂「資料怎麼被追蹤、怎麼上雲端」,才能聽懂 AI 在做什麼，判斷它做的事合不合理，而不是它說什麼就信什麼。
2. **資料夾命名要對得起來，多專案協作才不會迷路**——「資料夾命名 = 倉庫名稱」這個簡單原則,能省掉大量「這個改動要存去哪裡」的困惑,也才有辦法讓別人（或 AI agent）快速找到正確的東西。
3. **資安意識不是工程師專屬的責任**——複製貼上一個含 token 的網址,就可能不知不覺把鑑權碼洩漏出去。遇到不懂的設定問題，直接問 AI，不要裝懂跳過。
4. **重複性高的工作流程，值得花時間做成範本**——不需要先學會什麼厲害的工具，先把自己重複在做的事情誠實列出來，請 AI 幫忙整理成可以重複使用的結構，就是「做成範本」。
5. **判斷一個工具該複製客製化、還是該集中維護共用**——標準不是「現在內容是不是一樣」，而是「這個東西未來會不會因為不同案子而長成不一樣的版本」。

## 想從頭開始，建議從哪裡下手？步驟教學

如果看完這個系列，想自己也開始動手，不用一次到位，照這個順序，每一步都用對話請 AI agent 幫你做：

1. 先搞懂自己手上有什麼：
   > 「幫我看一下我 GitHub 帳號底下有哪些倉庫，本機資料夾跟這些倉庫對不對得起來」
2. 把資料夾整理成一致的結構：
   > 「幫我把這些資料夾統一搬到同一個地方，名稱改成跟倉庫一樣」
3. 檢查一次有沒有資安疑慮：
   > 「幫我檢查我手上的資料夾，remote 設定裡有沒有藏著帳密或 token」
4. 挑一件你最常重複做的事，試著做成範本：
   > 「我每次做 [某件重複的事] 都會做這些步驟：[列出來]，幫我整理成一份可以重複使用的範本」
5. 盤點一下手上有沒有「所有專案都要共用、但卻各自存了一份」的東西，把它獨立出來：
   > 「幫我看一下這幾個東西是不是該獨立成一個共用的地方，讓所有專案都能引用，而不是各自複製」

走完這五步，大概就重現了這整個系列的核心歷程。

## 常見問題（FAQ）

### PM 完全沒有工程背景，能學會 Claude Code 嗎？
可以。這個系列的作者就是非工程背景的 PM，整個學習過程不需要懂程式，只需要願意開口問 AI，一關一關走過去。Claude Code 的設計本來就是讓人用對話完成工作，指令不用背。

### 學 Claude Code 之前，需要先學什麼？
建議先看[第一篇](./01-pm-為什麼要懂一點-git-github.md)，搞懂 git 的基本邏輯（repo、commit、push）。有了這個基礎，後面整理資料夾、設定安全驗證、建立工作流範本都會更清楚在做什麼，而不只是照步驟操作。

### CLAUDE.md 和 agents/ 資料夾是 Claude Code 官方規定的格式嗎？
`CLAUDE.md` 是 Claude Code 會自動讀取的規則檔，這是官方機制。`agents/` 資料夾則是這個系列 PM 自己的命名習慣，不是強制規定——你可以叫它 `rules/`、`prompts/` 或任何喜歡的名字，重點是 AI agent 能被引用到那些說明文件。

### 這套工作流適合哪種 PM？
適合同時跑多個案子、需要頻繁產出文件（Spec、會議紀錄、簡報）、且想讓 AI 配合自己習慣的 PM。單一長期案子也適用，特別是在版本管理和文件累積方面。

### 照這個系列建好工作流之後，維護成本高嗎？
不高。CLAUDE.md 和 agents/ 是「寫一次、用很多次」的設定，只在流程改變或有新工具加入時才需要更新。日常工作不需要維護，只要照 SOP 步驟開案、工作結束前更新進度區塊即可。
