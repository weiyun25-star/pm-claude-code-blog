# 把重複的 PM 工作，變成一套我自己的規則

## 每接一個新案子，我都在重新發明輪子

每次接到新案子，流程幾乎都長得一樣：先搞懂背景、開需求訪談、把訪談紀錄整理成文件、寫 Spec、出 QA 測試計畫、最後 review 交付。但因為這些東西散落在不同檔案、不同習慣裡，每次開新案子，我幾乎都是憑印象重新來一次——這次訪談紀錄要問什麼？Spec 要包含哪些章節？上次那個 ROI 試算公式是怎麼算的？

後來我想：既然這些步驟根本是固定的，為什麼不把它寫成一套**自己的 SOP**，讓 AI agent 每次都照同一套規則跟我配合？這就是 `pm-workspace` 這個倉庫存在的原因——它不只是一份範本，而是我自己設計的一整套 PM 工作方法論。

## 第一塊：讓 AI 一開始就知道我是誰

在開始任何分析之前，我希望 AI 先知道「我是誰、我在意什麼」，所以寫了一份 `context/my-role.md`，裡面記錄：

- 我的職位、負責的產品線
- 主要責任：需求訪談、Spec 撰寫、跨部門溝通、數據分析
- 常合作的角色（工程師關注技術可行性、設計師關注體驗、主管關注 ROI）各自在意什麼
- 我的工作習慣：偏好先有結構再填內容、重要文件要 review 過才交出去、喜歡用數據支撐決策

這份檔案的規則很簡單：**任何分析或文件撰寫前，先讀這份檔案**。這樣 AI 給的建議才會貼合我實際的角色和習慣，不是一套放在哪個案子都通用的罐頭回答。

## 第二塊：把整套流程拆成 6 個步驟

核心是 `agents/new-project-sop.md`，把一個案子從接案到交付，拆成 6 個步驟：

```
git clone pm-workspace → [案子名稱]/
   ↓
填 BRIEF.md（案子背景）
   ↓
需求訪談 → 整理成 output/requirements/
   ↓
寫 Spec（架構/流程圖/工作項/功能邏輯/Wireframe） → output/specs/
   ↓
QA 測試計畫 → output/deliverables/
   ↓
Review → 交付
```

每一步都有對應的對話範例，不用自己想要怎麼問 AI，例如 Step 3 需求訪談整理：

> 「這是我的訪談紀錄，幫我整理成結構化需求，存到 output/requirements/interview-notes.md」

Step 4 寫 Spec：

> 「參考 @BRIEF.md @output/requirements/interview-notes.md @templates/spec-template.md，用 Socratic 問答幫我寫 Spec」

每一步該存到哪個資料夾，都寫得清清楚楚，不用每次重新決定檔案要放哪裡。

## 第三塊：一整組專門 agent，像是隨身帶著一群顧問

`agents/` 資料夾裡放的不是工具，更像是一群「專門角色」，需要的時候叫出來用：

| Agent | 做什麼 |
|------|--------|
| `spec-reviewer.md` | 用工程師／主管／使用者研究員三個角色分別 review Spec，找出各自視角會挑的問題 |
| `devil-advocate.md` | 魔鬼代言人，對決策找出最大弱點和被忽略的風險，目標不是否定，是讓決定更站得住腳 |
| `impact-estimator.md` | 用「Impact = 受影響用戶數 × 目前行為比例 × 預期提升 × 每次行為價值」公式，算出悲觀／現實／樂觀三種情境的 ROI |
| `competitive-research.md` | 競品分析，找出市場空缺跟我方該做/不做的方向 |
| `dev-estimate.md` | 把 Spec 拆成工作項，輔助估點 |
| `meeting-summarizer.md` | 整理會議紀錄 |
| `user-story-writer.md` | 需要詳細場景時，補一份 User Story |
| `wireframe-creator.md` | Spec 裡 Wireframe 區塊的格式規則 |

例如要對一個重大決策做壓力測試，不用自己想問題，直接說：

> 「參考 @agents/devil-advocate.md，幫我壓力測試這個決策：[貼上決策內容]」

它就會照固定的四步驟（列出核心假設 → 攻擊每個假設 → 找出被忽略的風險 → 提出反方案）跑一次分析。

## 第四塊：把我認同的方法論也寫進去

`knowledge/frameworks/` 放的是我認同、想讓 AI 在分析時優先採用的方法論，而不是讓它自己挑一套邏輯：

- **Impact Estimation**：算功能 ROI 時，永遠要給悲觀/現實/樂觀三個情境，不要只給一個數字
- **Rumelt's Strategy Kernel**：做策略分析時，用「診斷問題 → 訂出指導方針 → 連貫行動」三層框架，逼自己做出取捨，而不是寫一堆「提高營收」這種模糊到沒人會反對的目標

這塊的用意是：**讓 AI 用我認同的思考方式工作，而不是每次都要重新講一次「我想要的分析邏輯是這樣」**。

## 第五塊：讓自己隨時接得上次做到哪

BRIEF.md 最上面有一個「目前進度」區塊：

```
## 目前進度
- 目前階段：[對應 SOP 的 Step]
- 下一步：[要做什麼、需要誰提供什麼]
- 已產出文件：[路徑＋一句話說明]
- 最後更新：[日期]
```

規則是：每次工作階段結束前更新這個區塊，下次回來工作，先讀這個區塊再決定怎麼接續。這樣不管隔了多久沒碰這個案子，回來不用重新想「我做到哪了」。

## 步驟教學：怎麼用這套規則開始一個新案子

如果你也想試著建立自己的一套規則，用 pm-workspace 開新案子的方式很簡單，全程用對話完成：

1. 跟 AI agent 說：
   > 「幫我 clone pm-workspace 這個倉庫，建立一個叫 [案子名稱] 的新資料夾」

   它會幫你把整套結構（CLAUDE.md、BRIEF.md、agents、templates、knowledge、context、output）複製一份過去。
2. 接著說：
   > 「我接到一個新案子，幫我問清楚背景，問完後幫我填 BRIEF.md」

   它會照 `my-role.md` 跟 SOP 的設定，問你產品、目標用戶、時程資源等問題，並把答案填進 BRIEF.md。
3. 之後每進到下一個步驟（訪談整理、寫 Spec、QA、review），都直接說「參考 @agents/xxx.md，幫我做 xxx」，不用重新解釋規則，因為規則早就寫在那些檔案裡了。
4. 工作階段要結束前，說一句：
   > 「幫我更新 BRIEF.md 最上面的目前進度區塊」

   下次回來再說「幫我讀一下目前進度，接著做」，就能直接接續。

## 心得

自己設計一套工作流程規則,最大的差別不是「省了多少打字」，而是：**每個案子都用同一套思考邏輯被檢視過**——三角色 review、魔鬼代言人壓力測試、悲觀現實樂觀三情境 ROI，這些不是我每次想到才順手做，而是規則裡寫好「一定要做」，所以不會因為忙、因為趕時間就跳過。

對其他 PM 的建議：先別急著想「規則要多完整」，從你自己最常重複做的那一兩件事開始寫成一份 md 檔（像 `my-role.md` 或某個 agent），用幾次、覺得好用再慢慢補。規則是用出來的，不是一次設計完美的。
