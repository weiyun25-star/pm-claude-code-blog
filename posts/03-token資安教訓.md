# 我不小心把 GitHub Token 寫死在設定裡

## 那一行指令的輸出，讓我心跳漏了一拍

整理資料夾的過程中，我隨口請 AI agent 幫我看一下某個資料夾的 GitHub 連線設定。畫面跳出一行網址，我瞄了一眼，正常滑過去——直到 AI agent 接著說了一句：

「你的 git remote URL 裡直接嵌了一個 GitHub Personal Access Token，這個 token 明文存在設定裡。」

我愣了一下，倒回去重新看那行網址。果然，網址中間夾著一長串看起來像亂碼的字串，混在帳號跟網址之間，平常根本不會多看一眼。

它接著解釋：這串東西等於是「不需要密碼就能用你身份操作 GitHub 的鑑權碼」，任何能看到這個畫面、這個檔案的人，都可以拿著它假裝是你，去動你的倉庫。

我心裡咕嚇了一下，這段話看起來有點可怕。

## 過程：先搞懂它是什麼，再決定怎麼處理

### 第一步：搞懂 token 是什麼

Token 可以理解成「另一種形式的密碼」，差別是它通常是系統自動產生、用來讓工具（像 git）代替你跟 GitHub 溝通，不用每次都手動輸入帳密。問題在於：它**通常是明文存在設定檔裡**，一旦這個設定檔被看到、被截圖、被不小心上傳分享出去，效果就跟你的密碼洩漏一樣。

### 第二步：撤銷舊 token

確認風險之後，第一件事是去 GitHub 上把這個 token 撤銷掉，讓它馬上失效——就算之前已經洩漏，撤銷之後那串字串也變成廢紙,沒有任何作用。如果不知道在 GitHub 上哪裡操作，直接問 AI agent 就好：

> 「我要撤銷一個 GitHub token，要去哪裡操作？跟我說步驟」

它會告訴你要去哪個設定頁面、點哪裡，不用自己摸索。

### 第三步：改用更安全的驗證方式

撤銷舊 token 之後，不能讓 git 完全沒辦法驗證身份,所以接著請 AI agent 幫我設定一套更安全的方式：用 GitHub 官方工具 `gh auth login` 重新登入,並執行 `gh auth setup-git`，讓它接手處理身份驗證(這個機制叫 credential helper)。設定好之後，往後 push、pull 都會自動用安全的方式驗證，不需要把任何 token 明文寫在設定裡。

### 第四步：全面清查，不能只改一個地方

最讓我意外的是，這不是只有一個資料夾中獎。確認第一個資料夾後，我請 AI agent 順手檢查我手上其他幾個專案的資料夾，結果**陸續又挖出兩個資料夾**也用同樣方式嵌著 token——其中一個甚至是目前還有效、沒被撤銷的 token，一樣不該明文留著。每一個都用同一套方式清乾淨：把網址改成乾淨的版本，讓 credential helper 接手驗證。

## 步驟教學：怎麼檢查自己的設定裡有沒有藏著 token

如果你也想確認自己手上的資料夾有沒有同樣的問題，全程用對話請 AI agent 幫你檢查就好：

1. 先請它檢查一個資料夾：
   > 「幫我看一下這個資料夾的 git remote 設定，網址裡面有沒有藏著帳密或 token」
2. 如果有，請它說明風險，再決定下一步：
   > 「這個 token 會有什麼風險？我該怎麼處理？」
3. 確認要處理後，去 GitHub 上撤銷這個 token（這一步沒辦法用對話代勞，需要自己登入 GitHub 操作一次）。
4. 撤銷之後，回頭請 AI agent 幫你設定更安全的驗證方式：
   > 「我已經撤銷舊的 token 了，幫我設定一個更安全的方式，讓我之後 push、pull 不需要用 token」
5. 別只檢查一個資料夾就結束，請它順手檢查其他所有資料夾：
   > 「幫我檢查我手上其他專案的資料夾，看有沒有也藏著類似的 token，一併清乾淨」

走完這五步，可以確定自己手上所有的專案資料夾，都不再有任何 token 明文暴露在設定裡。

## 心得

這次的教訓讓我意識到：**資安意識不是工程師專屬的責任**。PM 雖然不寫程式，但只要會去串接帳號、設定工具、複製貼上別人給的指令，就有可能在不知不覺中把敏感資訊留在看不到的地方。

更重要的一點是：**只改一個地方是不夠的**。如果一個習慣（例如複製貼上含 token 的網址）造成了一次洩漏，很可能在其他地方也犯了同樣的習慣——這次全面清查下來，連自己都沒想到會有第三個地方中獎。下次遇到類似的設定問題，「順手檢查其他地方有沒有同樣的坑」會是一個值得養成的習慣。
