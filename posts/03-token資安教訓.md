# PM 我不小心把 GitHub Token 寫死在設定裡

## 那一行指令的輸出，讓我心跳漏了一拍

整理資料夾的過程中，我隨口請 AI agent 幫我看一下某個資料夾的 GitHub 連線設定。畫面跳出一行網址，我瞄了一眼，正常滑過去——直到 AI agent 接著說了一句：

「你的 git remote URL 裡直接嵌了一個 GitHub Personal Access Token，這個 token 明文存在設定裡。」

我愣了一下，倒回去重新看那行網址。果然，網址中間夾著一長串看起來像亂碼的字串，混在帳號跟網址之間，平常根本不會多看一眼。

它接著解釋：這串東西等於是「不需要密碼就能用你身份操作 GitHub 的鑑權碼」，任何能看到這個畫面、這個檔案的人，都可以拿著它假裝是你，去動你的倉庫。

我心裡咕嚇了一下，這段話看起來有點可怕。

## 發現 Token 洩漏後，第一步該怎麼做？

**處理流程是：先撤銷舊 token 讓它立刻失效、再設定安全的驗證方式取代 token、最後全面掃描所有專案資料夾確保沒有漏網之魚。**

### GitHub Token 是什麼？

Token 可以理解成「另一種形式的密碼」，差別是它通常是系統自動產生、用來讓工具（像 git）代替你跟 GitHub 溝通，不用每次都手動輸入帳密。問題在於：它**通常是明文存在設定檔裡**，一旦這個設定檔被看到、被截圖、被不小心上傳分享出去，效果就跟你的密碼洩漏一樣。

### 第二步：立即撤銷舊 token

確認風險之後，第一件事是去 GitHub 上把這個 token 撤銷掉，讓它馬上失效——就算之前已經洩漏，撤銷之後那串字串也變成廢紙,沒有任何作用。如果不知道在 GitHub 上哪裡操作，直接問 AI agent 就好：

> 「我要撤銷一個 GitHub token，要去哪裡操作？跟我說步驟」

它會告訴你要去哪個設定頁面、點哪裡，不用自己摸索。

### 第三步：改用更安全的驗證方式

撤銷舊 token 之後，不能讓 git 完全沒辦法驗證身份,所以接著請 AI agent 幫我設定一套更安全的方式：用 GitHub 官方工具 `gh auth login` 重新登入,並執行 `gh auth setup-git`，讓它接手處理身份驗證(這個機制叫 credential helper)。設定好之後，往後 push、pull 都會自動用安全的方式驗證，不需要把任何 token 明文寫在設定裡。

### 第四步：全面清查，不能只改一個地方

**一個帳號底下往往有多個資料夾都嵌著 token，光改第一個發現的不夠——必須掃描所有專案資料夾，每一個都清乾淨。**

最讓我意外的是，這不是只有一個資料夾中獎。確認第一個資料夾後，我請 AI agent 順手檢查我手上其他幾個專案的資料夾，結果**陸續又挖出兩個資料夾**也用同樣方式嵌著 token——其中一個甚至是目前還有效、沒被撤銷的 token，一樣不該明文留著。每一個都用同一套方式清乾淨：把網址改成乾淨的版本，讓 credential helper 接手驗證。

## 步驟教學：怎麼把 GitHub 驗證方式改安全？

如果你也想把 GitHub 的驗證方式換成更安全的設定，不用自己查資料、記指令，全程用對話請 AI agent 幫你做就好：

1. 先確認舊的 token 已經撤銷（如果有的話），這一步要自己登入 GitHub 操作，AI agent 沒辦法代勞。
2. 接著請它用官方工具重新登入：
   > 「幫我用 GitHub 官方的方式重新登入帳號」

   它會引導你完成登入流程，不需要再手動產生、複製任何 token。
3. 登入完成後，請它把這個帳號設定成 git 預設的驗證方式：
   > 「幫我把 git 設定成用剛剛登入的帳號自動驗證，以後 push、pull 不用再輸入任何 token」

   這一步做完，git 會自動透過剛剛登入的帳號驗證身份，不需要把任何東西明文寫進設定裡。
4. 請它幫你確認設定生效：
   > 「幫我檢查一下這個資料夾的 git remote 網址，確認裡面已經沒有任何 token 了」
5. 最後請它順手檢查其他專案資料夾，確保每一個都已經改成這套乾淨的驗證方式：
   > 「幫我檢查我手上其他專案的資料夾，把裡面殘留的 token 都清掉，改用剛剛設定好的驗證方式」

走完這五步，可以確定自己手上所有的專案資料夾，都不再有任何 token 明文暴露在設定裡。

## 常見問題（FAQ）

### GitHub Token 洩漏了，我的帳號會被盜嗎？
有風險。Token 等同「不需要密碼的鑑權通行證」，拿到它的人可以用你的身份對 GitHub 做任何你有權限的操作。立即撤銷（Revoke）是第一步，撤銷後該 token 馬上失效，即使之前已經洩漏也沒用了。

### 怎麼確認 git remote URL 裡有沒有夾 token？
請 AI agent：「幫我看一下這個資料夾的 git remote 設定」，它會列出 URL。如果網址格式是 `https://[一串字串]@github.com/...`，那串字串就是 token，需要清掉。

### credential helper 是什麼？設定之後還需要 token 嗎？
Credential helper 是讓 git 自動用安全方式驗證身份的機制，設定好之後不需要在 URL 裡嵌 token。往後 push、pull 都透過這個機制驗證，不會把任何憑證明文寫在設定檔裡。

### 撤銷 token 之後，現有的 git 操作會壞掉嗎？
撤銷後，還在用這個 token 的 git remote URL 都會驗證失敗。所以撤銷的同時，要把那些 URL 改乾淨、並設定好 credential helper，這樣 push/pull 才能繼續正常運作。

### PM 不寫程式，為什麼也要注意 token 資安？
因為 PM 也會用 AI agent 設定工具、複製貼上別人給的指令，這些操作都可能產生含 token 的設定。資安風險不只發生在「寫程式」的時候，凡是串接帳號、設定工具，都可能留下漏洞。
