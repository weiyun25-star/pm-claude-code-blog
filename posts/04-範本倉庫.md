# 一個案子一個資料夾不夠用——用「範本倉庫」管理重複流程

## 我猜了一個做法，結果意外猜對了大半

資料夾整理好之後，我開始想下一個問題：以後接到新案子，難道每次都要重新生出一套 BRIEF、訪談紀錄、Spec 的資料夾結構？感覺一定有更好的方式。

我自己先猜了一個做法，跑去問 AI agent：「我以後如果要新起專案，是不是要先在 repo 底下建資料夾，然後把 pm-workspace 的東西 clone 下來？」

我以為這只是我自己亂猜的土法煉鋼，結果 AI agent 回我：方向是對的，只是少了一個關鍵步驟——它直接拿我手上既有的 `insurance` 這個案子去對照 `pm-workspace`，兩邊的 git 紀錄一比，答案就出來了。

## 過程：用 git log 找到證據

它比對了兩個資料夾最早的幾筆紀錄，結果一模一樣：

```
insurance 最早的紀錄：
8be0115 Initial PM workspace setup
c345429 Restructure: add output folder, remove projects folder...
1d7e9e8 Add dev-estimate agent and estimates folder

pm-workspace 最早的紀錄：
8be0115 Initial PM workspace setup
c345429 Restructure: add output folder, remove projects folder...
1d7e9e8 Add dev-estimate agent and estimates folder
```

完全一樣的前幾筆——這就證明了 `insurance` 這個案子，當初真的是從 `pm-workspace` clone 出來的，後面才開始累積自己案子專屬的內容（像是「Fill BRIEF for 台灣人壽會員任務分級制案」這種屬於這個案子的紀錄）。

完整的流程是這樣：

1. **clone 一份 pm-workspace**：到新資料夾，這時候本機的內容、commit 歷史，跟 pm-workspace 一模一樣，帶著現成的 `agents/`、`templates/`、`BRIEF.md` 結構。
2. **在 GitHub 上開一個全新的空倉庫**：用這個案子的名字（例如 `insurance`），這一步是我猜漏的地方——我以為 clone 完直接用就好，沒想到要另外開一個新倉庫。
3. **把這個資料夾的 remote 改成新倉庫的網址**：不然之後做的修改，會一直想推回 `pm-workspace` 本身，而不是這個案子自己的倉庫。
4. **push 上去**，把模板結構帶到新倉庫，之後照 `CLAUDE.md` 的規則開始填 `BRIEF.md`、累積這個案子專屬的內容。

## 步驟教學：怎麼用範本倉庫開一個新案子

如果你也想試著用同樣的方式管理重複性的工作，全程用對話就能完成：

1. 跟 AI agent 說：
   > 「幫我把 pm-workspace clone 一份，建立一個叫 [案子名稱] 的新資料夾」
2. 接著請它確認/建立對應的新倉庫，並把連線改過去：
   > 「幫我在 GitHub 上開一個新的倉庫叫 [案子名稱]，然後把這個資料夾的 remote 改成連到這個新倉庫」
3. 確認連線正確後，請它推上去：
   > 「幫我把這個資料夾 push 上去」
4. 之後開始這個案子，直接說：
   > 「我接到一個新案子，幫我問清楚背景，填到 BRIEF.md」

   它會照範本裡寫好的規則，問你產品、目標用戶、時程資源等問題。

如果不確定一個既有資料夾是不是真的是從範本 clone 出來的，也可以請它幫你查證：

> 「幫我看一下這個資料夾最早的幾筆 commit，跟 pm-workspace 比對一下是不是同一個起點」

## 心得

這篇最大的收穫，不是學會了「範本倉庫」這個概念，而是體會到：**自己猜的做法，拿出來跟 AI agent 核對一次，往往能補上自己沒想到的細節**。我猜對了大方向（clone 範本），但漏了「要另外開新倉庫、改 remote」這個關鍵步驟——如果沒有核對，大概會一路把後面的案子內容都不小心推回 pm-workspace 本身,搞砸範本本身。

對其他 PM 的建議：工作流程裡「重複性高」的部分（像是每次接案都要做的固定動作），值得花時間做成一份範本，讓 AI agent 接手執行固定步驟，而不是每次都憑印象重新來一次。
