# PM 的 AI agent 工作流？用「範本倉庫」管理重複流程

## 那種「好像有用、但又不踏實」的感覺

用 AI 工作這段時間，有一種感覺一直甩不掉：明明感覺自己在進步——從一開始只會單純問答，到後來可以請它幫我直接產出檔案，照理說應該越來越順手才對。但每次坐下來要開始做事，心裡還是有一股說不上來的不踏實。

後來想明白了，問題不是「會不會用」，是**整體的工作流程根本沒有建立起來**。我可以叫 AI 幫我寫一段文字、整理一份筆記，但只要碰到 PM 真正該產出的那些東西——Spec 要怎麼透過 AI 產出？User Story 又該怎麼寫才對？這些念頭一直在腦中打轉，卻沒有一個清楚的答案。每次都是憑當下的心情決定要問什麼、怎麼問，做完也說不出個所以然，像是在原地打轉，而不是真的往前走。

## 既然都聽到要做成範本了，那我就自己做一個

這段時間我一直聽到身邊的人在講「重複的事要做成 skill」「這個流程應該寫進 CLAUDE.md，不要每次重新講一遍」。一開始聽不太懂，只覺得這些詞好像很厲害，但跟我一個 PM 有什麼關係？

後來資料夾整理好之後，我開始想下一個問題：以後接到新案子，難道每次都要重新生出一套 BRIEF、訪談紀錄、Spec 的資料夾結構？每次都要重新跟 AI 解釋一次「我要的格式是這樣」？這時候才突然意識到——這不就是大家說的「重複的事要做成規則」嗎？

每次接到新案子，流程幾乎都長得一樣：先搞懂背景、開需求訪談、把訪談紀錄整理成文件、寫 Spec、出 QA 測試計畫、最後 review 交付。但因為這些東西散落在不同檔案、不同習慣裡，每次開新案子，我幾乎都是憑印象重新來一次——這次訪談紀錄要問什麼？Spec 要包含哪些章節？上次那個 ROI 試算公式是怎麼算的？

於是我沒有再等，直接動手請 AI agent 幫我把自己的工作流程整理成一份**範本倉庫**——`pm-workspace`。不是去找現成的工具來套，而是照著我自己接案子的習慣，一步一步把該有的東西生出來，讓 AI agent 每次都照同一套規則跟我配合。

## 範本結構長什麼樣子

整理完之後，這份範本大概是這樣：

```
pm-workspace/
├── CLAUDE.md       全局規則：語言、檔案存放規則、開始新案子前要做什麼
├── BRIEF.md        案子背景空白範本（目標用戶、成功定義、利害關係人...）
├── agents/         專門用的 agent（review、估點、會議整理...），裡面也放著一份 new-project-sop.md，是整套接案流程的 SOP
├── templates/      Spec、會議紀錄的空白模板
├── knowledge/       跟 AI 討論累積下來的方法論與心得
├── context/        my-role.md，讓 AI 知道我是誰、工作習慣是什麼
└── output/         專門放 AI 產出的資料，依類型分裝到下面四個子資料夾
    ├── requirements/
    ├── specs/
    ├── meeting-notes/
    └── deliverables/
```

重點不是資料夾本身，而是**每個資料夾都對應一條明確的規則**：訪談紀錄固定存哪裡、Spec 固定存哪裡、開始新案子前要先讀哪個檔案、工作階段結束前要更新哪個區塊。這些規則寫進 `CLAUDE.md` 之後，AI agent 每次跟我配合，都照同一套邏輯走，不用我每次重新講一遍。

建好範本之後，下一個問題是：**新案子要怎麼套用這份範本？** 答案不是把範本內容複製貼上，而是用 git 的方式整份「clone」過去，這樣新案子一開始就帶著完整的結構跟規則，後面才開始累積自己案子專屬的內容：

1. **clone 一份 pm-workspace**：到新資料夾，這時候內容跟 pm-workspace 一模一樣，帶著現成的 `agents/`、`templates/`、`BRIEF.md` 結構。
2. **在 GitHub 上開一個全新的空倉庫**：用這個案子的名字（例如 `insurance`）。
3. **把這個資料夾的 remote 改成新倉庫的網址**：不然之後做的修改，會一直想推回 `pm-workspace` 本身，而不是這個案子自己的倉庫。
4. **push 上去**，把範本結構帶到新倉庫，之後照 `CLAUDE.md` 的規則開始填 `BRIEF.md`、累積這個案子專屬的內容。

## 範本裡實際放了哪些規則

光有資料夾結構還不夠，真正讓這套範本好用的，是裡面藏的五塊規則：

### 第一塊：讓 AI 一開始就知道我是誰

在開始任何分析之前，我希望 AI 先知道「我是誰、我在意什麼」，所以寫了一份 `context/my-role.md`，裡面記錄我的職位、主要責任、常合作的角色各自在意什麼、我的工作習慣（偏好先有結構再填內容、重要文件要 review 過才交出去、喜歡用數據支撐決策）。

規則很簡單：**任何分析或文件撰寫前，先讀這份檔案**。這樣 AI 給的建議才會貼合我實際的角色和習慣，不是一套放在哪個案子都通用的罐頭回答。

### 第二塊：把整套流程拆成 6 個步驟

核心是 `agents/new-project-sop.md`，把一個案子從接案到交付，拆成 6 個步驟：

```
clone pm-workspace → [案子名稱]/
   ↓
填 BRIEF.md（案子背景）
   ↓
需求訪談 → 整理成 output/requirements/
   ↓
寫 Spec（架構/流程圖/工作項/功能邏輯/Wireframe） → output/specs/
   ↓
QA 測試計畫 → output/deliverables/
   ↓
Review → 交付
```

每一步都有對應的對話範例，不用自己想要怎麼問 AI，例如需求訪談整理：

> 「這是我的訪談紀錄，幫我整理成結構化需求，存到 output/requirements/interview-notes.md」

寫 Spec：

> 「參考 @BRIEF.md @output/requirements/interview-notes.md @templates/spec-template.md，用 Socratic 問答幫我寫 Spec」

每一步該存到哪個資料夾，都寫得清清楚楚，不用每次重新決定檔案要放哪裡。

### 第三塊：一整組專門 agent，像是隨身帶著一群顧問

`agents/` 資料夾裡放的不是工具，更像是一群「專門角色」，需要的時候叫出來用：

| Agent | 做什麼 |
|------|--------|
| `spec-reviewer.md` | 用工程師／主管／使用者研究員三個角色分別 review Spec，找出各自視角會挑的問題 |
| `devil-advocate.md` | 魔鬼代言人，對決策找出最大弱點和被忽略的風險，目標不是否定，是讓決定更站得住腳 |
| `impact-estimator.md` | 用「Impact = 受影響用戶數 × 目前行為比例 × 預期提升 × 每次行為價值」公式，算出悲觀／現實／樂觀三種情境的 ROI |
| `competitive-research.md` | 競品分析，找出市場空缺跟我方該做/不做的方向 |
| `dev-estimate.md` | 把 Spec 拆成工作項，輔助估點 |
| `meeting-summarizer.md` | 整理會議紀錄 |
| `user-story-writer.md` | 需要詳細場景時，補一份 User Story |
| `wireframe-creator.md` | Spec 裡 Wireframe 區塊的格式規則 |

例如要對一個重大決策做壓力測試，不用自己想問題，直接說：

> 「參考 @agents/devil-advocate.md，幫我壓力測試這個決策：[貼上決策內容]」

它就會照固定的四步驟（列出核心假設 → 攻擊每個假設 → 找出被忽略的風險 → 提出反方案）跑一次分析。

### 第四塊：把跟 AI 討論學到的東西，累積成我的知識庫

`knowledge/frameworks/` 這塊的用意，不是一次寫好就放著不動，而是**每次跟 AI 溝通討論時，只要學到一個我覺得值得留下來的觀念或結論，就請它幫我記到這裡**。久而久之，這個資料夾變成一個會持續累積的「我跟 AI 一起想過的東西」，下次遇到類似的問題，AI 可以直接拿之前討論出的結論來用，不用每次都從零開始討論一次。

目前裡面已經累積了幾個討論出來、覺得值得留下來的方法論：

- **Impact Estimation**：算功能 ROI 時，永遠要給悲觀/現實/樂觀三個情境，不要只給一個數字
- **Rumelt's Strategy Kernel**：做策略分析時，用「診斷問題 → 訂出指導方針 → 連貫行動」三層框架，逼自己做出取捨，而不是寫一堆「提高營收」這種模糊到沒人會反對的目標

這塊真正的價值是：**讓 AI 用我們一起討論、累積下來的思考方式工作，而不是每次都要重新講一次「我想要的分析邏輯是這樣」**。

### 第五塊：讓自己隨時接得上次做到哪

BRIEF.md 最上面有一個「目前進度」區塊：

```
## 目前進度
- 目前階段：[對應 SOP 的 Step]
- 下一步：[要做什麼、需要誰提供什麼]
- 已產出文件：[路徑＋一句話說明]
- 最後更新：[日期]
```

規則是：每次工作階段結束前更新這個區塊，下次回來工作，先讀這個區塊再決定怎麼接續。這樣不管隔了多久沒碰這個案子，回來不用重新想「我做到哪了」。

## 步驟教學：怎麼用範本倉庫開一個新案子

這裡分成兩個階段：**第一階段只會做一次**（建範本），**第二階段是每次接新案子都會重複的動作**（用範本開案子）。

### 階段一：建立範本（只做一次）

1. 先跟 AI agent 討論你目前的工作流程，把每個步驟講清楚：
   > 「我每次接案子大概會做這些事：[列出你的固定流程]，幫我整理一下，看這個流程有沒有漏掉什麼步驟」

   先討論清楚，不要急著叫它生檔案——這一步是要先把你腦中模糊的習慣，整理成一套有順序、講得出道理的流程。
2. 流程確認沒問題之後，才請它把這套流程寫成一份 SOP 檔案，跟對應的範本資料夾結構：
   > 「這套流程確定了，幫我寫成一份 SOP 檔案，並整理成一份可以重複使用的範本資料夾結構」

   這個範本（包含 SOP 檔案）之後會一直被重複使用，不用每次重新討論一遍。

### 階段二：用範本開新案子（每次接案都這樣做）

3. 要開新案子時，直接請它把範本 clone 一份：
   > 「幫我把這份範本 clone 一份，建立一個叫 [案子名稱] 的新資料夾」
4. 接著請它確認/建立對應的新倉庫，並把連線改過去：
   > 「幫我在 GitHub 上開一個新的倉庫叫 [案子名稱]，然後把這個資料夾的 remote 改成連到這個新倉庫」
5. 確認連線正確後，請它推上去：
   > 「幫我把這個資料夾 push 上去」
6. 接著請它讀取範本裡的 SOP 文件，照步驟開始：
   > 「參考 @agents/new-project-sop.md，幫我開始這個新案子」

   它會自己依照 SOP 寫好的步驟走——先問你產品、目標用戶、時程資源等問題，填進 BRIEF.md，不用你再重新講一次規則是什麼。
7. 之後每進到下一個步驟（訪談整理、寫 Spec、QA、review），都直接說「參考 @agents/xxx.md，幫我做 xxx」，規則早就寫在那些檔案裡了。
8. 工作階段要結束前，說一句：
   > 「幫我更新 BRIEF.md 最上面的目前進度區塊」

   下次回來再說「幫我讀一下目前進度，接著做」，就能直接接續。

## 心得

先說一下「工作流」是什麼：簡單講，就是**一件事從開始到結束，固定會經過的那幾個步驟，以及每個步驟之間怎麼接起來**。對 PM 來說，工作流就是「接到案子 → 搞懂背景 → 訪談 → 寫文件 → 交付」這一整條路徑——這篇想說的「整體工作流程沒建立起來」，指的就是這條路徑一直沒有被講清楚、寫下來，每次都是憑當下心情決定下一步要做什麼。

這篇最大的收穫，是體會到「做成範本」這件事，門檻其實沒有想像中高——不需要先學會什麼厲害的工具或框架，**先把自己手上重複在做的事情，誠實地列出來，再請 AI agent 幫你整理成一份可以重複使用的結構，就已經是「做成範本」了**。

更深一層的價值不是「省了多少打字」，而是：**每個案子都用同一套思考邏輯被檢視過**——三角色 review、魔鬼代言人壓力測試、悲觀現實樂觀三情境 ROI，這些不是我每次想到才順手做，而是規則裡寫好「一定要做」，所以不會因為忙、因為趕時間就跳過。

對其他 PM 的建議：先別急著想「規則要多完整」，從你自己最常重複做的那一兩件事開始寫成一份 md 檔（像 `my-role.md` 或某個 agent），用幾次、覺得好用再慢慢補。規則是用出來的，不是一次設計完美的。
