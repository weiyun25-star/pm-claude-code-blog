# PM 為什麼要懂一點 Git／GitHub？

## 情境切入

整理工作資料夾的過程中，我請 AI agent 幫忙看一下我手上的專案，它隨口提到一句：「你這個資料夾的 remote 連的是 resume-grace 這個 repo。」

我當下完全愣住——repo？remote？這些是什麼？

後來我直接問了它：「repo git github 意思」，得到了一個我覺得對 PM 來說很好懂的解釋，整理如下。

## 過程：三個詞，三個角色

- **Git**：一套版本控制軟體（工具本身）。負責記錄檔案每次的修改歷史，可以追蹤誰改了什麼、何時改的，也能回到之前的版本。
- **Repo（repository，倉庫）**：一個「被 Git 管理的專案資料夾」。裡面除了你的檔案，還多了一個隱藏的 `.git` 資料夾，存放所有歷史記錄。
- **GitHub**：一個雲端平台，用來存放、備份、分享 Git repo。本機的 repo 透過指令把資料送上 GitHub，這樣就算電腦壞了，東西也還在雲端；也方便跟別人協作。

關係可以這樣理解：

```
Git    = 工具（怎麼記錄版本）
Repo   = 被 Git 管理的專案（資料本體 + 歷史記錄）
GitHub = 存放 repo 的雲端服務（讓 repo 不只活在你電腦裡）
```

當我去查自己 GitHub 帳號底下到底有什麼，發現竟然已經累積了 7 個倉庫：pm-workspace、insurance、learnhub、coffee-personality-quiz、resume-grace、weeklybm_tool、practice。每一個都是我曾經做過的案子或專案，只是我從來沒有意識到「它們各自是一個獨立被 Git 追蹤的資料夾」。

## 心得

PM 不需要會寫程式，但理解「資料怎麼被追蹤、怎麼上雲端」這套邏輯，其實很重要，原因有兩個：

1. **跟工程師溝通會更順**：當工程師說「我推到某個 branch」「這個改動還沒 merge」，至少知道他們在說什麼狀態。
2. **跟 AI agent 一起工作會更安心**：AI agent 操作的所有檔案、所有指令，背後幾乎都跟 git/repo 有關。知道這套邏輯，才能判斷 agent 做的事情合不合理，而不是完全黑箱地相信它。

下一篇，我會分享發現自己手上有 7 個倉庫之後，怎麼把散落各處、亂七八糟的本機資料夾，重新整理成一套一看就懂的結構。
