# PM 為什麼要懂一點 Git／GitHub？

## 那句「你也要學一下 git 啦」，意外點醒了我

那一天，我坐在位子上操作著 Claude Code，想說讓它幫我整理一下手上亂糟糟的專案資料夾。旁邊的工程同事路過，瞄了一眼我的畫面，笑著說了一句：

「你也要學一下 git 啦。」

我愣了一下，嘴上應了聲「好啊」，心裡其實在想：git 不是工程師的東西嗎？我一個 PM，到底要懂到什麼程度？

結果沒過多久，我請 AI agent 幫忙看一下我手上的專案，它隨口提到一句：「你這個資料夾的 remote 連的是 resume-grace 這個 repo。」

我又愣住了——repo？remote？這些又是什麼？

## 過程：三個詞，三個角色

- **Git**：一套版本控制軟體（工具本身）。負責記錄檔案每次的修改歷史，可以追蹤誰改了什麼、何時改的，也能回到之前的版本。
- **Repo（repository，倉庫）**：一個「被 Git 管理的專案資料夾」。裡面除了你的檔案，還多了一個隱藏的 `.git` 資料夾，存放所有歷史記錄。
- **GitHub**：一個雲端平台，用來存放、備份、分享 Git repo。本機的 repo 透過指令把資料送上 GitHub，這樣就算電腦壞了，東西也還在雲端；也方便跟別人協作。

關係可以這樣理解：

```
Git    = 工具（怎麼記錄版本）
Repo   = 被 Git 管理的專案（資料本體 + 歷史記錄）
GitHub = 存放 repo 的雲端服務（讓 repo 不只活在你電腦裡）
```

當我去查自己 GitHub 帳號底下到底有什麼，發現竟然已經累積了 7 個倉庫：pm-workspace、insurance、learnhub、coffee-personality-quiz、resume-grace、weeklybm_tool、practice。每一個都是我曾經做過的案子或專案，只是我從來沒有意識到「它們各自是一個獨立被 Git 追蹤的資料夾」。

## 為什麼資料要用 Git？對 PM 有什麼好處？

- **版本歷史**：每次 commit 都是一個「存檔點」，可以回頭看「這份文件三天前是什麼樣子」「這個改動是誰、什麼時候做的」。PM 常常要追蹤「規格是哪個版本」「會議紀錄改過幾次」，Git 把這件事自動記錄下來，不用自己手動存 `v1`、`v2`、`v2_final`、`v2_final_真的最終版`。
- **不怕弄丟、改壞**：改錯了可以退回去之前的版本，不會像直接覆寫檔案一樣，一旦存檔就回不去。
- **跟工程師協作有共同語言**：知道「這個 commit 改了什麼」「push 了沒」，溝通會更順。
- **自動備份到雲端**：push 上 GitHub 後，就算這台電腦掛了，東西還在。

## 資料夾裡的 `.git` 是什麼？

前面提到，被 Git 追蹤的資料夾裡會有一個隱藏的 `.git` 資料夾。它是 Git 拿來「記帳」的資料庫——裡面存著這個資料夾從第一次建立（`git init`／`git clone`）到現在，每一次 commit 的完整歷史、每個檔案每個版本的內容、跟 GitHub 的連線設定（remote）等等。**平常不需要打開它、也不該手動改裡面的東西**，它就是 Git 自己在背後維護的記錄本。一個資料夾有沒有 `.git`，就決定了它是不是一個被 Git 追蹤的「repo」。

有個地方一開始很容易誤會：請 AI agent 幫忙產生檔案，並不會自動生出 `.git`。我自己就遇過——先請它幫我整理出一份文件，過程中突然想到「欸，這個資料夾或檔案應該要有版本控制吧」，結果一看，裡面根本沒有 `.git`。AI agent 產生檔案，就只是單純的「建立檔案」動作,跟「這個資料夾有沒有被 Git 追蹤」完全是兩件事，它不會自己偷偷幫你做這件事。

而且這裡其實藏著**兩個分開的決定**，常常被當成同一件事：

1. **這個資料夾要不要變成 git 倉庫**：也就是要不要請 AI agent 幫你執行 `git init`，讓它開始被 Git 追蹤、可以 commit。
2. **要不要在 GitHub 上開一個對應的倉庫，把它連過去**：這是另外一回事——本機可以單純只有 git 倉庫、先 commit 存版本，完全不連 GitHub 也沒問題；要不要備份到雲端、要不要讓別人也看得到，是之後可以再決定的第二步。

兩者不是綁在一起、也不會互相觸發。直接用對話講清楚要做到哪一步就好，例如：

> 「幫我把這個資料夾變成一個 git 倉庫」——只會做第 1 步
>
> 「幫我把這個資料夾變成一個 git 倉庫，然後在 GitHub 上開一個新倉庫連過去」——兩步一起做

## 檔案沒有上 GitHub，要怎麼看版本差異？

這裡也很容易誤會：版本差異是不是要等檔案上了 GitHub 才看得到？其實不用。所有歷史紀錄都存在本機的 `.git` 裡，跟有沒有 push 完全無關——只要這個資料夾是 git 倉庫、有 commit 過，差異就已經被記下來了。

而且不是打開某個「特定檔案」去看，版本差異不會存在某一份檔案裡讓你直接點開，而是要透過 Git 把這些紀錄「叫出來看」。一樣不用記指令，直接用對話問 AI agent 就好，例如：

> 「幫我看一下這個檔案的修改紀錄」——它會列出這個檔案每一次 commit 的時間、說明
>
> 「這個檔案這次的改動，跟上一版差在哪裡？」——它會列出具體哪幾行被改了、改成什麼

這也是為什麼前面會提醒「不要手動亂動 `.git`」——因為所有版本差異的紀錄都存在那裡面，亂動很容易把歷史搞壞。

## `git commit` 跟 `git push` 是什麼關係？會自動 push 嗎？

這是我一開始最搞不清楚的地方——以為「存檔」跟「上傳」是同一件事。其實是**分開的兩個動作**，不會自動串連：

1. **`git commit`**：把目前的修改「存一個版本」，但這個存檔**只存在自己電腦的 `.git` 裡**，GitHub 上完全看不到、雲端不會更新。
2. **`git push`**：把本機已經 commit 的這些版本，**送上 GitHub**，這時候雲端才會同步更新。

簡單比喻：`commit` 像是「存檔」，`push` 像是「上傳雲端硬碟」。可以一直存檔（commit 很多次）放在自己電腦，直到想分享/備份的時候，才執行一次 `push` 把這些存檔都送上去。**不會自動 push**——沒有人（或 AI agent）主動下 `git push` 指令，東西就一直留在本機，GitHub 上不會有任何變化。

## 步驟教學：怎麼檢查自己手上的資料夾是不是一個 git repo

如果你也想知道自己電腦裡某個資料夾，到底是不是被 Git 追蹤的 repo，可以照著做一次：

1. 打開終端機（Terminal／命令提示字元），用 `cd` 移動到你想檢查的資料夾，例如：
   ```
   cd /workspace/project/insurance
   ```
2. 輸入以下指令，看看這個資料夾裡有沒有隱藏的 `.git` 資料夾：
   ```
   ls -la
   ```
   如果列出的清單裡看到一個叫 `.git` 的資料夾，代表這裡**是**一個 git repo；沒有的話，它就只是一般的資料夾。
3. 如果確認是 repo，再輸入以下指令，看看這個 repo 連到 GitHub 上的哪個倉庫：
   ```
   git remote -v
   ```
   會看到類似這樣的輸出：
   ```
   origin  https://github.com/你的帳號/倉庫名稱.git (fetch)
   origin  https://github.com/你的帳號/倉庫名稱.git (push)
   ```
   這就是這個資料夾「真正對應」的 GitHub 倉庫——以後在這個資料夾做的任何 commit、push，都會送到這個網址。
4. 如果想看自己 GitHub 帳號底下總共有幾個倉庫，且本機沒有安裝對應工具，最直接的方式是打開瀏覽器，登入 GitHub，到自己的 Profile 頁面點「Repositories」，就會列出全部清單。

照著做一次之後，你大概就能掌握「我手上這些資料夾，到底哪些是被 Git 管理的，分別連到哪個倉庫」。

## 步驟教學：實際做一次 commit 跟 push，體會兩者的差異

這裡有個好消息：在 AI 的世界裡，commit 跟 push 其實**不用自己下指令**，直接用對話跟 AI agent 說就可以了。背後它還是會執行 `git status`、`git add`、`git commit`、`git push` 這些步驟，只是你不需要記指令，只要說清楚你想要它做什麼。

在一個已經是 repo 的資料夾裡（前面步驟確認過有 `.git` 的那種），試著修改一個檔案，然後照下面的方式跟 AI agent 對話一次：

1. 改完一個檔案後，先問它目前狀態，可以說：
   > 「幫我看一下這個資料夾有哪些檔案被改過」

   它會幫你執行 `git status`，列出哪些檔案被改了、哪些是新檔案。
2. 接著請它存一個版本，可以說：
   > 「幫我把這個改動 commit，說明是『調整了 XXX』」

   這一步完成後，**GitHub 上還看不到任何變化**——這個版本目前只存在本機。
3. 確認雲端還沒更新：打開瀏覽器看一下這個倉庫在 GitHub 上的頁面，會發現剛剛的修改還沒出現。
4. 接著請它推上去，可以說：
   > 「幫我 push 上去」或「commit 完幫我直接推上 GitHub」

   這時候再重新整理 GitHub 頁面，就會看到剛剛的修改出現了。

走過這幾步，會很明顯感覺到：commit 跟 push 是兩個獨立的動作，就算全部交給 AI agent 用對話完成，commit 完也不代表雲端就更新了，一定要再請它 push 才會真正同步上去。也因為這樣，當你看到 agent 說「我幫你 commit 好了」，可以多問一句「那有 push 上去嗎」，確認雲端是不是真的同步了。

**PM學習小心得**

PM 不需要會寫程式，但理解「資料怎麼被追蹤、怎麼上雲端」這套邏輯，其實很重要，原因有兩個：

1. **跟工程師溝通會更順**：當工程師說「我推到某個 branch」「這個改動還沒 merge」，至少知道他們在說什麼狀態，不會只是點頭如搗蒜假裝聽懂。

   來翻譯一下工程師常掛在嘴邊的這兩句：
   - **「我推到某個 branch」**：branch（分支）可以想成「在同一份文件上開的不同副本，各自獨立修改，互不干擾」。主線通常叫 `main`，工程師另外開一個叫 `feature-login` 的分支去改登入功能，這樣不會動到主線目前穩定的版本。「推到某個 branch」就是把改動 push 到這個獨立副本上，不是直接動主線——說白了，就是先在自己的小宇宙裡改，還沒打算驚動大家。
   - **「這個改動還沒 merge」**：merge（合併）就是「把某個分支上的修改，正式併回主線」。還沒 merge，代表它目前只活在那個分支裡，main 上看到的版本還是舊的——翻成 PM 語言就是：**這功能還沒有真的「上線」，先別急著跟老闆說做完了。**

2. **跟 AI agent 一起工作會更安心**：AI agent 操作的所有檔案、所有指令，背後幾乎都跟 git/repo 有關。知道這套邏輯，才能判斷 agent 做的事情合不合理，而不是完全黑箱地相信它。
