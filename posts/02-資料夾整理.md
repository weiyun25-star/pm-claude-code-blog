# 專案資料夾亂成一團？讓 AI Agent 幫你整理

## 那句「這個文件你不是早就做好了嗎」，把我問倒了

某次跨部門會議上，主管臨時問了一句：「你之前那個保險案子的需求文件呢？先丟給我看一下。」

我嘴上說「好，我找一下」，手卻在電腦裡開始亂翻。`project` 資料夾、`complete-course` 資料夾、還是哪個案子順手存的？翻了快十分鐘，找到的還不確定是不是最新版。會議室裡的沉默，比文件本身還難找。

那一刻我才意識到：問題不是「我沒做這份文件」，是我**連自己手上有什麼、放在哪裡，都說不清楚**。

回去之後，我順手請 AI agent 幫我查一下，自己 GitHub 帳號底下到底有多少倉庫。結果列出來又嚇了一跳：`pm-workspace`、`insurance`、`learnhub`、`coffee-personality-quiz`、`resume-grace`、`weeklybm_tool`——N 個。

更尷尬的是，我本機的資料夾完全對不起來。盤點下來，問題大概有這幾種：

- **專案完全沒出現在任何看得懂的地方**：有些倉庫對應的本機資料夾，根本不知道躲去哪了。
- **被塞進「雜物資料夾」、巢狀好幾層**：雜物資料夾裡面偷偷塞了好多專案資料。
- **資料夾名稱跟倉庫名稱不一致**：`insurance` 這個倉庫——對，就是主管問的那個保險案子——本機對應的資料夾其實藏在 `project/insurance` 裡；`learnhub` 則藏在 `complete-course/learnhub`。
- **同一個專案出現兩份不同大小寫、內容也不完全一樣的資料夾**：假設專案叫 `shop-app`，結果同時存在 `shop-app` 跟 `SHOP-APP` 兩個資料夾，沒人說得清哪一份才是最新的。

而大小寫不同的重複，特別容易被忽略，因為它牽涉到一個容易被忽略的技術細節：**Linux 系統的檔案路徑是「大小寫敏感」的**，`shop-app` 跟 `SHOP-APP` 在系統眼裡是完全不同的兩個資料夾，可以同時存在、互不衝突，系統不會跳出警告提醒你「這跟另一個資料夾很像，是不是搞錯了」。但換到 Windows 或 macOS 預設的檔案系統，大小寫是不敏感的，這兩個名字會被當成同一個資料夾，根本開不出兩份。也就是說：**系統技術上允許的事，不代表對人來說是合理的**——這正是當時兩份內容不同、名字幾乎一樣的資料夾，可以並存好幾個月卻沒人發現的原因。

我心裡只有一個念頭：我永遠都找不到我要調整的檔案。

## 過程：先盤點，再決定怎麼歸位

**整理資料夾的核心邏輯分三步：用 AI 列出 GitHub 上的倉庫清單、對比本機資料夾哪裡對不起來、最後統一搬到命名一致的位置。全程用對話完成，不需要自己翻資料夾或下指令。**

這整個過程，我沒有自己一個一個資料夾去翻、自己動手搬移，而是**全程用對話請 AI agent 幫我做**。整理的邏輯很單純，分三步，每一步都是一句話請它執行：

1. **盤點 GitHub 上實際有哪些倉庫**——我說「幫我查一下我 GitHub 帳號底下有哪些倉庫」，它直接列出全部，當作對照表的「正解」。
2. **盤點本機資料夾，找出哪些其實是被 Git 追蹤的獨立倉庫**——我說「幫我檢查這幾個資料夾，哪些是 git 倉庫、各自連到 GitHub 上的哪個倉庫」，它用前一篇學到的方法（檢查有沒有 `.git`、看 remote 連到哪裡）一個一個確認，整理出對照表。結果發現好幾個倉庫的本機資料夾，藏在跟倉庫名稱完全不相關的位置。
3. **規劃一個統一的位置，資料夾名稱直接對應倉庫名稱**——確認對照表沒問題後，我說「幫我把這些資料夾統一搬到 repos/ 底下，改成跟倉庫名稱一樣」，它就把整理結果實際搬移到 `/workspace/repos/<倉庫名稱>` 底下，例如 `repos/insurance`、`repos/learnhub`、`repos/pm-workspace`。

整個過程我沒有打過一行指令，只是不斷確認它盤點出來的對照表對不對、要不要調整，最後一句話請它執行搬移。

過程中還遇到一個意外：有個資料夾怎麼搬都搬不動，系統一直回報「裝置忙碌中」。後來才發現，它其實不是普通資料夾，而是一個獨立的磁碟掛載點——這種情況就不該硬搬，維持原地就好。這也是請 AI agent 一起整理的好處：它能順手檢查清楚「為什麼搬不動」，而不是直接用更暴力的方式硬上。

還有一些資料夾完全沒有對應的 GitHub 倉庫（例如裡面放著某個專案的 PM 文件、截圖、架構圖），這種「孤兒資料夾」我沒有照單全收地刪除或合併，而是請 AI agent 把每個資料夾的內容、跟其他資料夾的差異整理成白話說明給我聽，我再依據這些說明決定保留、刪除，還是先不處理。

遇到看起來很像、又不確定能不能直接處理的資料夾時（例如懷疑是重複的兩份），AI agent 不會自己決定刪除或合併，而是先比較兩邊內容的差異，告訴我哪裡不一樣，再問我要怎麼處理。這時候也可以反過來問它：「如果我把這個搬走/刪掉，會有什麼影響？」讓它先講清楚後果，自己再做決定，而不是看它建議就直接照做。

## 步驟教學：怎麼盤點自己的資料夾跟倉庫對不對得起來

如果你也想檢查自己的資料夾是不是也有同樣的問題，可以用對話請 AI agent 幫你做：

1. 先請它列出 GitHub 上的倉庫清單：
   > 「幫我看一下我 GitHub 帳號底下有哪些倉庫」
2. 接著請它檢查本機某個資料夾，是不是其中一個倉庫的本機副本：
   > 「幫我檢查一下這個資料夾是不是 git 倉庫，如果是的話，它連到 GitHub 上的哪個倉庫」
3. 如果手上有好幾個看起來亂糟糟、巢狀很深的資料夾，可以一次請它整理：
   > 「幫我盤點這幾個資料夾底下，有哪些是獨立的 git 倉庫，分別對應 GitHub 上的哪個倉庫，幫我整理一個對照表」
4. 確認對照表之後，再請它幫你搬到統一的位置：
   > 「幫我把這些對應到 GitHub 倉庫的資料夾，搬到統一的資料夾底下，並改成跟倉庫名稱一樣」
5. 如果有搬不動的資料夾，先別急著要求它用更強硬的方式處理，問清楚原因：
   > 「這個資料夾搬不動，幫我查一下原因」
6. 如果有兩個資料夾看起來很像、不確定是不是重複，先別急著刪：
   > 「這兩個資料夾內容差在哪裡？」
   確認差異之後，再決定要不要處理，也可以多問一句：
   > 「如果我把這個搬走（或刪掉），會有什麼影響？」

走完這六步，手上的資料夾會跟 GitHub 上的倉庫一一對應，看資料夾名稱就知道它連到哪裡。

**PM學習小心得**

**「資料夾命名 = 倉庫名稱」是讓多專案協作不迷路的最簡單原則，對人、對同事、對 AI agent 都適用。**

這篇想強調的不是「資料夾要整理得多漂亮」，而是一個更現實的理由：

> PM 要橫跨好幾個案子追蹤進度、找文件、交接給別人。如果連自己的資料夾都理不清楚，要怎麼讓別人（甚至 AI agent）幫你快速找到正確的東西？

PM 常常是團隊裡唯一同時碰好幾個專案的人——工程師通常只專注自己負責的那個倉庫，但 PM 要同時追蹤多個案子的進度、找文件、隨時準備交接。如果連自己的資料夾結構都說不清楚，不管是請同事接手、還是請 AI agent 幫忙，都會卡在「先搞懂你的資料夾長怎樣」這一步，浪費掉本來可以拿來做事的時間。

「資料夾命名 = 倉庫名稱」這個原則很簡單，但效果立竿見影：以後不管是你自己、同事、還是 AI agent，看到資料夾名稱就知道它對應哪個倉庫，不會再卡在「這個改動要 push 去哪裡」這種小事上。

## 常見問題（FAQ）

### AI agent 整理資料夾，會不會誤刪東西？
好的 AI agent 在整理前會先列出對照表、請你確認，不會自己決定刪除或合併。遇到有疑問的資料夾，先問「這兩個資料夾差在哪裡？」確認之後再決定，不要叫它直接刪。

### 本機資料夾名稱跟 GitHub 倉庫名稱不一樣，一定要改嗎？
不是強制，但強烈建議。命名一致的好處是：你自己、同事、AI agent 看到資料夾名稱就知道它對應哪個倉庫，不用每次都去查。命名不一致則是未來混亂的根源。

### 整理資料夾之前需要先備份嗎？
如果資料夾都已經是 git 倉庫（有 `.git` 且有 commit 過），本身就有完整的版本歷史，搬移路徑不會影響 git 記錄。如果有資料夾**沒有被 git 追蹤**，搬移前請先確認裡面沒有唯一的重要文件。

### 資料夾搬移後，push 還能正常運作嗎？
能。git 的 remote 設定存在 `.git/config` 裡，搬移資料夾路徑不會改變 remote 設定，push 一樣正常。

### 大小寫不同的兩個資料夾（例如 `shop-app` 和 `SHOP-APP`）怎麼處理？
先請 AI agent 比較兩個資料夾的內容差異，確認哪一份是最新的。確認後把舊的刪掉或歸檔，保留一份並統一命名小寫。不要猜——先比內容，再決定。
